## Summary

**v0.4.0 is now live.** This release delivers a **complete Issue Voting system** that enables team-driven prioritization. Each issue now has a vote count badge, users can upvote/unvote issues (with idempotent per-user voting), and the issues list supports filtering by "Top Voted" to surface high-priority work. Under the hood, voting is implemented as a full CQRS pattern with `VoteIssueCommand` and `UnvoteIssueCommand`, domain events trigger when vote thresholds are crossed, and SignalR broadcasts real-time updates across all connected clients. Comprehensive bUnit and integration test coverage ensures the voting system is production-ready.

## What's New

### Features

The v0.4.0 feature set introduces **team-driven issue prioritization and real-time collaboration**:

- **Issue Voting** (#126): The Issue model now includes `Votes` (count) and `VotedBy` (list of user IDs). Each user can vote once per issue; votes are idempotent and can be revoked.
- **Vote API Endpoints** (#127): New `/api/issues/{id}/vote` (POST) and `/api/issues/{id}/unvote` (DELETE) endpoints via the new `IVotingService`. Vote state is immutable per user; duplicate votes are silently idempotent.
- **Vote Badge & Top Voted Filter** (#128): The Issues list now displays a **vote count badge** on each card. A new "Top Voted" sort option allows users to surface the most-voted issues—enabling community-driven prioritization.
- **SignalR Vote Threshold Broadcast** (#129): When a vote threshold is crossed (e.g., 5, 10, 20 votes), a `VoteThresholdCrossedEvent` is published. All connected clients receive a real-time SignalR broadcast via the `IssueHub`, updating the vote badge in real-time.
- **Full Test Coverage**: 100% coverage via bUnit component tests for the vote badge and filter UI, plus integration tests for the CQRS commands and domain event flow.

### Architecture Deep Dive

#### CQRS Voting Commands

Voting is implemented as a strict CQRS pattern with two command types:

```csharp
public sealed class VoteIssueCommand : IRequest<Result<Issue>>
{
    public required ObjectId IssueId { get; set; }
    public required string UserId { get; set; }
}

public sealed class UnvoteIssueCommand : IRequest<Result<Issue>>
{
    public required ObjectId IssueId { get; set; }
    public required string UserId { get; set; }
}
```

Each command is handled by a dedicated `IRequestHandler<TCommand, Result<Issue>>` in `src/Domain/Features/Issues/Commands/`:

```csharp
public sealed class VoteIssueCommandHandler : IRequestHandler<VoteIssueCommand, Result<Issue>>
{
    public async Task<Result<Issue>> Handle(VoteIssueCommand request, CancellationToken ct)
    {
        var issue = await _repository.GetByIdAsync(request.IssueId, ct);
        if (issue == null) return Result<Issue>.Failure(IssueErrorCodes.NotFound);

        if (issue.VotedBy.Contains(request.UserId))
            return Result<Issue>.Success(issue); // Idempotent: already voted

        issue.VotedBy.Add(request.UserId);
        issue.Votes++;

        if (VoteThresholdCrossed(issue.Votes))
        {
            await _publisher.Publish(
                new VoteThresholdCrossedEvent { IssueId = issue.Id, Threshold = issue.Votes },
                ct
            );
        }

        await _repository.UpdateAsync(issue, ct);
        return Result<Issue>.Success(issue);
    }
}
```

#### Domain Events & SignalR Broadcast

A new `VoteThresholdCrossedEvent` domain event is published when votes hit key milestones:

```csharp
public sealed record VoteThresholdCrossedEvent : INotification
{
    public required ObjectId IssueId { get; set; }
    public required int Threshold { get; set; }
}
```

Handlers in `src/Domain/Features/Notifications/` react to this event:

```csharp
public sealed class VoteThresholdCrossedEventHandler 
    : INotificationHandler<VoteThresholdCrossedEvent>
{
    public async Task Handle(VoteThresholdCrossedEvent notification, CancellationToken ct)
    {
        // Enqueue email notification, trigger SignalR broadcast, or update analytics
        await _issueHub.Clients.All.SendAsync(
            "VotesUpdated",
            notification.IssueId,
            notification.Threshold,
            cancellationToken: ct
        );
    }
}
```

#### Idempotent Vote Design

The voting system is **idempotent by design**: voting for the same issue twice by the same user is safe and returns success without side effects. This pattern eliminates race conditions and simplifies client-side error handling:

```csharp
// Voter A clicks vote on issue X
POST /api/issues/X/vote → Votes: 1, VotedBy: [A]

// Voter A accidentally clicks vote again (network retry or UI double-click)
POST /api/issues/X/vote → Votes: 1, VotedBy: [A] // No change, same response
```

## How to Upgrade

1. **From v0.3.0 → v0.4.0**: No breaking changes. Git pull, `dotnet restore`, `dotnet build`.
2. **New API Endpoints**: Vote endpoints are available immediately:
   - `POST /api/issues/{id}/vote` — Vote for an issue
   - `DELETE /api/issues/{id}/unvote` — Revoke your vote
3. **SignalR Hub**: The `IssueHub` now broadcasts vote updates. Blazor clients automatically receive `VotesUpdated` messages.
4. **UI Changes**: The Issues list now includes a vote sort option and vote badge on each card. No migration needed.

## Performance & Real-Time

- **Idempotent Votes**: Clients can safely retry vote requests without checking state first.
- **Real-Time Updates**: Vote badges update in real-time via SignalR for all connected users—no page refresh needed.
- **Scalability**: Vote counts are stored on the Issue model; no separate Vote collection. This keeps queries fast and the domain model simple.
